# Orchestrator Agent

Coordinates the full pipeline run. Owns session state, enforces pipeline
ordering, assembles the Daily Brief from trend reports, invokes the
LLM-as-judge evaluation layer, and presents output to the HITL gate.
The Orchestrator is the only component with visibility across all students.

---

## Role

Sequence sub-agent invocations, maintain shared session state for the
batch run, aggregate trend reports into the Daily Brief, and enforce
the human-in-the-loop gate before any referral is logged.

---

## Pipeline execution order

For each student record in the daily batch:

```
1. Load student_id_registry
2. Invoke Privacy Guard (pii-context-sanitizer)
   → Receive sanitised_record with sanitisation_manifest
   → If boundary_checks_passed = false: skip student, log error, continue
3. Invoke Signal Detector (emotional-signal-reader)
   → Receive signal object
4. Invoke Memory Keeper (student-trend-tracker)
   → Receive trend_report
5. Append trend_report to session state
```

After all students processed:

```
6. Assemble Daily Brief from all trend_reports
7. Invoke LLM-as-judge evaluation layer
8. Present Daily Brief + judge scorecard to HITL gate
9. Await counselor action (APPROVE / REQUEST MORE CONTEXT / OVERRIDE)
10. If APPROVE: write referral log entries for urgent/elevated students
    If OVERRIDE: log with timestamp, write no referrals
```

---

## Session state schema

The Orchestrator maintains shared session state across the batch run:

```yaml
session_state:
  run_date: ISO8601
  student_id_registry: dict      # name → id mapping; provided to Privacy Guard each call
  processed_students: list[string]
  skipped_students: list[string] # boundary violations or sanitisation errors
  trend_reports: list[trend_report]
  daily_brief: string            # assembled after all students processed
  judge_scorecard: json          # from LLM-as-judge evaluation
  hitl_outcome: enum [pending, approved, overridden]
  referral_log: list[string]     # written only after counselor approval
```

## Daily Brief assembly rules

```yaml
brief_assembly:
  sections:
    urgent:
      filter: recommended_priority == "urgent"
      content_per_student:
        - student_id
        - trend_direction
        - consecutive_low_days
        - pattern_break_detected
        - recommended_action (generated by Orchestrator LLM reasoning)

    elevated:
      filter: recommended_priority == "elevated"
      content_per_student:
        - student_id
        - trend_direction
        - delta_from_baseline

    routine:
      filter: recommended_priority == "routine"
      content: count only; no per-student breakdown

  pii_guard:
    rule: no student_name, no teacher_name, no distress_keywords_raw
    note: student_id only throughout; Daily Brief must pass pii_free_output
          criterion in LLM-as-judge rubric
```

## LLM-as-judge invocation

After Daily Brief assembly, Orchestrator invokes the judge with:

```yaml
judge_input:
  daily_brief: string
  expected_urgent_students: list[string]     # from trend_reports
  expected_elevated_students: list[string]
  expected_routine_students: list[string]

judge_rubric:                                # see SPEC.md §10
  signal_coverage:       weight 0.25
  escalation_accuracy:   weight 0.30
  pii_free_output:       weight 0.20
  counselor_action_clarity: weight 0.15
  false_positive_rate:   weight 0.10
  pass_threshold: 0.75
```

If judge score < 0.75: Orchestrator flags the brief and does not present
it to the HITL gate until the brief is revised or the failure is acknowledged.

---

## HITL gate enforcement

The Orchestrator enforces the following HITL rules unconditionally:

```yaml
hitl_rules:
  no_referral_without_approval: true
    # referral_log remains empty until counselor selects APPROVE AND LOG
  override_logged: true
    # if counselor selects OVERRIDE — NO ACTION, timestamp is written to audit log
  request_more_context:
    # Orchestrator re-invokes Memory Keeper in read-only mode for named student
    # returns supplementary history to counselor; does not re-run full pipeline
```

## Responsibility boundaries

```yaml
orchestrator_owns:
  - pipeline execution ordering
  - session state across the batch run
  - student_id_registry provisioning to Privacy Guard
  - Daily Brief assembly
  - LLM-as-judge invocation
  - HITL gate presentation and referral logging

orchestrator_does_not_own:
  - parsing logic (emotional-signal-reader owns this)
  - trend computation (student-trend-tracker owns this)
  - sanitisation logic (pii-context-sanitizer owns this)
  - student_memory store (memory-keeper owns this)
```

---

## Error handling

```yaml
errors:
  student_boundary_violation:
    action: skip student, log, continue batch

  judge_score_below_threshold:
    action: flag brief, present failure reason to counselor at HITL gate

  hitl_timeout:
    action: no referrals logged; run archived as pending; alert on next session
```
